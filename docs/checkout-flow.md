# Оформление покупки: фронт-флоу (абонемент и пробное)

> **Назначение.** UX оформления покупок на фронте: шаги, конструктор, выбор ребёнка,
> вход на страницу. Ивент не описывается (анонимная «быстрая касса», без шагов).
> **Статус.** Актуальный — единственный авторитет по UX оформления покупок.
> Оговорка: пробные занятия в реализацию пока не вошли (этап отложен, см.
> «Статус реализации» в `api-core-contracts.md`) — разделы про пробное сохранены
> как проект под этот этап.
> **Связанные документы.** HTTP-контракты — `api-core-contracts.md` (Сценарий 4);
> авторизация и `redirectFrom` — `auth-flow.md`.

---

## 1. Общая модель: страница под авторизацией, 2 шага + редирект

**Вся страница оформления (и абонемента, и пробного) — под авторизацией.** Гость на
неё не попадает: клик «Оформить абонемент» / «Купить пробное» уводит на вход и
возвращает назад через `redirectFrom` (§4). **Корзины и сохранения состояния нет** —
ушёл со страницы, собираешь заново.

Оплата — **не шаг визарда**, а выход из него (редирект на ЮКассу). Выбор ребёнка —
не отдельный экран, а часть подтверждения. Итого два содержательных шага:

```
[на страницу попал уже авторизованным]
Шаг 1. Конструктор покупки
        │
        ▼
Шаг 2. Подтверждение заказа
        • выбор ребёнка (+ инлайн «добавить ребёнка»)
        • ревью состава и суммы
        • кнопка «Оплатить»
        │
        ▼
   редирект на ЮКассу
```

Каждый лишний экран визарда роняет конверсию — поэтому выбор ребёнка слит с ревью,
а оплата вынесена за пределы шагов. Наглядность цены для незалогиненных закрывает
публичный «Калькулятор абонемента» на главной (§6).

---

## 2. Шаг 1 — Конструктор

### Абонемент

- Сетка слотов кружков; тык по слоту добавляет его в корзину, тык повторно — убирает.
- **Переключатель недели — только для просмотра доступности мест** (места плавают
  неделя к неделе из-за пробных и масок). На состав абонемента выбор недели **не влияет**.
- **Семантика слотов абсолютная.** В корзину падает паттерн «ПН 16:00» (`schedule_id`),
  а не календарная дата. Абонемент действует месяц и сам развернётся в 4 ближайшие даты
  по каждому слоту. В UI это надо подписать явно («места показаны на неделю X, абонемент
  действует месяц с первого занятия»), иначе пользователь решит, что записывается на
  конкретную неделю.
- **Тариф не выбирается — выводится из числа слотов** (`project-context.md` §6:
  1 слот=4000, 2=7000, 3=10000, 4=11000, 5=12000, >5=безлимит 15000 ₽). Показывать
  живой пересчёт: «Выбрано 2 слота → тариф „8 занятий“ → 7 000 ₽». Можно подсказывать
  пороги выгоды.
- Сумма на фронте — **только для UX**. Авторитет цены — бэкенд (см. §5).
- Пересекающиеся по времени слоты в одной корзине — предупреждать прямо здесь.

### Пробное

- Выбор кружка → из списка дейттаймов выбор **конкретного календарного** слота
  (дата+время), напр. «Вт 17.06.2026 14:00–15:00, свободно 4/6».
- В отличие от абонемента, дата здесь настоящая, не паттерн.

## 3. Шаг 2 — Подтверждение заказа

Один экран, на нём:
- **Выбор ребёнка** — селект из своих детей + инлайн «добавить ребёнка» (если детей нет —
  тоже инлайн, без ухода со страницы).
- **Ревью** — состав покупки (слоты/кружок+дата), ребёнок, итоговая сумма/тариф.
- **«Оплатить»** → создание транзакции → редирект на `payment_url` ЮКассы.

---

## 4. Вход на страницу и `redirectFrom`

Страница оформления целиком за авторизацией — auth-gate стоит **на входе**, не между
шагами. Корзины/персиста нет: ушёл — собираешь заново.

Кнопки «Оформить абонемент» / «Купить пробное» (на главной, в каталоге) для
неавторизованного уводят на вход, передавая целевой маршрут в query:

```
клик → /login?redirectFrom=/checkout/subscription
     → OTP-вход (auth-flow.md); redirectFrom тащится через ВСЕ экраны входа (email → код)
     → после verify фронт читает redirectFrom и редиректит на страницу оформления
```

Авторизован — сразу на страницу, без логина.

**Безопасность `redirectFrom` (обязательно).** Это вектор open redirect. Принимать
**только внутренние относительные маршруты из белого списка** (`/checkout/subscription`,
`/checkout/trial`). Любой абсолютный URL, протокол, `//host`, неизвестный путь —
отбрасывать на дефолтный маршрут. Иначе `redirectFrom=https://evil.example` после
входа унесёт пользователя на фишинг. Хранить на время входа — в query роута логина
или в `sessionStorage` (это цель-строка, не корзина).

## 5. Серверная истина перед оплатой

Между «гость собрал корзину» и «нажал Оплатить» проходит время — места разбирают,
лимиты всплывают. Клиентским расчётам доверять нельзя. Поэтому на входе в Шаг 2
(и при сабмите) — финальная серверная проверка. Коды ошибок — `api-core-contracts.md` §0.3:

| Проверка | Ошибка при провале | Когда показать |
| -------- | ------------------ | -------------- |
| Места ещё свободны (`free > 0`) | `409 NO_AVAILABLE_SEATS` | подсветить отвалившийся слот |
| По пробному: у ребёнка ещё не было пробного по этому кружку | `409 TRIAL_LIMIT_EXCEEDED` | **на выборе ребёнка**, до оплаты |
| Сумма пересчитана сервером | — | клиентская сумма только для показа |

Лимит пробного проверять реактивно **в момент выбора ребёнка**: выбрал ребёнка →
бэкенд говорит «по этому кружку пробное уже было» → блокируем кнопку. Не давать
дойти до ЮКассы и словить отказ там.

## 6. Калькулятор абонемента на главной (публичный)

Поскольку страница оформления под авторизацией, наглядность цены для незалогиненных
даёт публичный блок на главной. Это **не покупка**, а прикидка: «N слотов → тариф →
сумма» по тем же правилам, что конструктор (§2, `project-context.md` §6).

- Данные тарифов — публичный `GET /api/v1/public/plans/` (`api-core-contracts.md`),
  расчёт суммы на клиенте.
- Свободные места не нужны (записи нет) — только числа и цена.
- Кнопка «Оформить» из калькулятора ведёт в обычный auth-gated флоу через `redirectFrom` (§4).
