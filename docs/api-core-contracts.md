# API-контракты ядра «Улица Радости» — проектное решение

> **Назначение.** HTTP-контракты ядра: URI, payload, ответы, ошибки, вебхуки, слои DRF.
> **Статус.** Актуальный — авторитет по HTTP-контрактам, с отклонениями реализации
> (см. блок «Статус реализации» ниже). Живая схема — `GET /api/schema/` (drf-spectacular).
> Поведение входа и покупок — в `auth-flow.md` и `checkout-flow.md`.
> **Связанные документы.** `schema.md`, `project-context.md`, `schema-audit.md`,
> `schema-refactoring.md`, `public-forms-design.md`; легаси — `api-contracts-legacy.md`.
>
> Внешние стандарты: формат ошибок — **RFC 9457 «Problem Details for HTTP APIs»**;
> идемпотентность — IETF draft **`Idempotency-Key`** + паттерн Stripe; вебхуки —
> рекомендации ЮКассы (проверка IP + перезапрос статуса).

---

## Статус реализации

Ядро реализовано по этому контракту: OTP-вход, чекаут абонемента с Idempotency-Key,
вебхук ЮКассы, публичная витрина, ЛК, RFC 9457. Три осознанных отклонения:

- **Пробные занятия** (`POST /checkout/trial`) — не реализованы, этап отложен
  владельцем проекта; контракт сохранён на будущее.
- **Оплата ивентов** (`POST /checkout/event`) — вместо онлайн-оплаты сделана гостевая
  регистрация `POST /api/v1/public/events/{id}/register/` со статусом
  `PENDING_PAYMENT` и оплатой на месте; бронь освобождается свипером через 30 минут.
- **Журнал учителя** (`/staff/*`) — реализован в админ-панели (django-unfold), а не
  как отдельное API; см. `admin.md`. Отдельные staff-эндпоинты не понадобились.

## 0. Общие соглашения (cross-cutting)

Эти правила обязательны для **всех** эндпоинтов, включая публичные формы из
`public-forms-design.md`. Единообразие — половина удобства клиента (Nuxt 4).

### 0.1. Версионирование и базовый путь
`/(api)/v1/...`. Версия в URI (не в заголовке) — проще кэшировать, отлаживать
curl-ом и читать логи. Группировка по аудитории: `public/`, `auth/`, `me/`,
`checkout/`, `webhooks/`, `staff/`.

### 0.2. Заголовки

| Заголовок (запрос)         | Назначение |
| -------------------------- | ---------- |
| `Authorization: Bearer …` *(или cookie, см. 1.4)* | Сессия родителя/преподавателя |
| `Idempotency-Key: <uuid4>` | Обязателен для `POST` checkout и отметок посещаемости (Шаг 2) |
| `Content-Type: application/json` | Везде, кроме загрузки файлов |

| Заголовок (ответ)          | Назначение |
| -------------------------- | ---------- |
| `Content-Type: application/problem+json` | Только для ошибок (RFC 9457) |
| `Retry-After: <сек>`       | На `429` и на cooldown OTP |
| `X-RateLimit-Remaining`    | Сколько запросов осталось в окне |

### 0.3. Единый формат ошибок (RFC 9457) — на всю систему

Любая ошибка — `application/problem+json`. Базовые поля по RFC 9457 + наши расширения:

```json
{
  "type": "https://api.ypa-center.ru/problems/no-available-seats",
  "title": "Нет свободных мест",
  "status": 409,
  "detail": "В группе «Кружок мышления, ПН 16:00» не осталось мест.",
  "instance": "/api/v1/checkout/subscription",
  "code": "NO_AVAILABLE_SEATS",
  "errors": [
    { "field": "schedule_ids[0]", "code": "SEAT_TAKEN", "message": "Мест нет" }
  ]
}
```

- `type` — стабильный URI-идентификатор класса проблемы (можно завести страницу-описание).
- `code` *(расширение)* — машинный enum для `switch` на фронте, не привязанный к локали.
- `errors[]` *(расширение)* — пофайловые ошибки валидации (DRF-стиль, но завёрнутый).

**Каталог бизнес-ошибок** (единый для ядра и форм):

| HTTP | `code`                     | Когда |
| ---- | -------------------------- | ----- |
| 400  | `MALFORMED_REQUEST`        | Битый JSON, неизвестный тип продукта |
| 401  | `AUTH_REQUIRED`            | Нет/протух токен (покупка абонемента/пробного гостем) |
| 403  | `FORBIDDEN_RESOURCE`       | Доступ к чужому ребёнку/абонементу (приватность) |
| 404  | `NOT_FOUND`                | Слот/ивент/ребёнок не существует |
| 409  | `NO_AVAILABLE_SEATS`       | Нет мест в слоте/ивенте |
| 409  | `TRIAL_LIMIT_EXCEEDED`     | Уже было пробное по этому кружку |
| 409  | `SUBSCRIPTION_EXPIRED`     | Абонемент просрочен (списание/возврат фишки) |
| 409  | `IDEMPOTENCY_KEY_REUSED`   | Ключ переиспользован с другим телом запроса |
| 422  | `VALIDATION_ERROR`         | Поля не прошли валидацию (формат, обязательность) |
| 429  | `RATE_LIMITED`             | Превышен троттлинг / cooldown OTP |

Граница **422 vs 409**: `422` — данные синтаксически приняты, но семантически
невалидны (плохой email, пустой `schedule_ids`); `409` — данные валидны, но
конфликтуют с текущим состоянием мира (мест нет, лимит исчерпан). Это снимает
вечный спор «какой код на бизнес-конфликт».

### 0.4. Деньги и время
Деньги — `integer` в копейках (как в `schema.md`). Время в ответах — ISO 8601 с TZ
(`2026-06-13T16:00:00+07:00`); абсолютные слоты — `day_of_week (0..6)` + `"HH:MM"`.

## Шаг 1. Контракты по сценариям

### Сценарий 1 — Беспарольная аутентификация (OTP по email)

> Поведенческая модель авторизации (режимы доступа, регистрация = первый вход,
> онбординг, удалённая теневая регистрация) — в `auth-flow.md`. Здесь только HTTP-контракт.
> Изменение схемы под OTP (`magic_tokens.code` + `attempts_count` вместо magic-link
> `token`) зафиксировано в `schema-refactoring.md`.

**Шаг A — запрос кода**

```
POST /api/v1/auth/otp/request
{ "email": "olga.e@example.com" }
```

Ответ — **всегда `202 Accepted`, независимо от того, есть ли такой email** (анти-энумерация:
нельзя по ответу понять, зарегистрирован ли адрес):

```json
{ "status": "sent", "resend_available_in": 60, "code_ttl": 300 }
```

- `resend_available_in` — cooldown в секундах: фронт рисует таймер «Отправить повторно через 0:59».
- На повторный запрос до истечения cooldown — `429` + `Retry-After`.

**Шаг B — верификация**

```
POST /api/v1/auth/otp/verify
{ "email": "olga.e@example.com", "code": "418302" }
```

Успех — `200`, выдача сессии (см. 1.4). Ошибки: `422 VALIDATION_ERROR` (код не 6 цифр),
`409`/`401 OTP_INVALID` (неверный/истёкший код), `429` после N неудачных попыток.

**Архитектурные ответы:**

1. **Защита от флуда** (см. также `public-forms-design.md §2.1`):
   - cooldown на отправку — 60 с на email (счётчик в Redis);
   - не более 5 запросов кода / час на email и на IP (`ScopedRateThrottle`);
   - не более 5 попыток ввода на один код, иначе код сжигается (`attempts_count` в таблице).
2. **Cooldown-таймер клиенту** — через тело (`resend_available_in`) и заголовок
   `Retry-After` на `429`. Фронт не считает сам — бэкенд авторитетен по времени.
3. **Хранение сессии — рекомендация: JWT в `HttpOnly; Secure; SameSite=Lax` cookie.**
   Обоснование под Nuxt 4 (SSR): токен в `localStorage`/`Bearer` доступен JS → крадётся
   при любой XSS. `HttpOnly` cookie недоступна JS и автоматически уходит при SSR-фетче.
   Цена — нужна CSRF-защита (двойной токен/`SameSite`). Если команда сознательно
   выбирает Bearer ради простоты и mobile-переиспользования — допустимо, но это
   осознанный размен безопасности на простоту, а не дефолт.
   Django-сессии отвергнуты: лишний серверный стейт, хуже для будущего mobile-клиента.
4. **Когда создаётся `Parent`** — *только при первом успешном `verify`*, не на
   `request` (детали — `auth-flow.md` §5). После `verify`: профиль пуст → онбординг
   (регистрация), профиль заполнен → кабинет (логин). Неподтверждённых профилей не
   существует: до verify в БД пишется лишь OTP-запись с TTL.

### Сценарий 2 — Динамическая сетка расписания

```
GET /api/v1/public/schedule?week_start=2026-06-15
```

`week_start` — понедельник запрашиваемой недели (по умолчанию текущая). Это **query-параметр,
а не часть пути**: неделя — это фильтр над одним ресурсом «расписание», а не отдельный
ресурс. Бэкенд проецирует абсолютные слоты на даты этой недели, накладывает маски и
считает места — клиент получает готовую к рендеру сетку.

**Response (200)** — плоский список, фронт раскидывает по колонкам дней:

```json
{
  "week_start": "2026-06-15",
  "week_end": "2026-06-21",
  "slots": [
    {
      "schedule_id": 106,
      "date": "2026-06-15",
      "day_of_week": 0,
      "start_time": "16:00",
      "end_time": "17:00",
      "activity": { "id": 1, "name": "Кружок мышления", "slug": "kruzhok-myshleniya" },
      "teacher": { "id": 2, "full_name": "Яков Леонидович Мордвинов" },
      "room": { "id": 3, "name": "Кабинет 2" },
      "capacity": { "max": 6, "taken": 4, "free": 2 },
      "is_rescheduled": false,
      "is_cancelled": false,
      "override": null
    },
    {
      "schedule_id": 106,
      "date": "2026-06-22",
      "start_time": "18:00", "end_time": "19:00",
      "is_rescheduled": true,
      "override": { "original_start_time": "16:00", "reason": "Перенос" }
    }
  ]
}
```

**Архитектурные ответы:**

- **Проекция слотов на даты** — бэкенд берёт `[week_start, week_end]`, для каждого
  активного слота вычисляет `date = week_start + day_of_week`. Никаких будущих дат в БД нет.
- **Маски** — на каждую дату подтягиваются активные `schedule_mask`: `CANCELLATION`
  → `is_cancelled: true` (слот остаётся в выдаче, помеченный, чтобы UI показал «отменено»);
  `RESCHEDULE` → подменяются время/день/кабинет, `is_rescheduled: true`, в `override`
  кладётся оригинал.
- **Свободные места** — `taken = активные REGULAR-записи + неистёкшие HELD-брони +
  пробные на эту неделю` (бронь до оплаты учтена, см. `schema-audit §1.1`). **Все три
  значения (`max/taken/free`) считает бэкенд** — фронт не вычитает.
- Структура **денормализована**: `activity/teacher/room` вложены прямо в слот, фронту
  не нужны доп. запросы и склейки.

### Сценарий 3 — Личный кабинет (Dashboard)

**Решение: НЕ один тяжёлый запрос, а несколько независимых ресурсов.** Обоснование:

- разные части дашборда имеют разную «скорость порчи» (профиль статичен, лента занятий
  меняется ежедневно) → разный кэш;
- лента будущих занятий пагинируется и тяжела — нельзя держать профиль её заложником;
- частичная деградация: упал расчёт ленты — профиль и абонементы всё равно отрисуются.

| Ресурс | Эндпоинт | Что отдаёт |
| ------ | -------- | ---------- |
| Профиль + дети | `GET /api/v1/me/profile` | данные родителя, массив `children` (writable nested, см. Шаг 3) |
| Абонементы | `GET /api/v1/me/subscriptions` | активные абонементы с балансом по слотам |
| Лента занятий | `GET /api/v1/me/upcoming?weeks=4&child_id=&cursor=` | агрегированная хронолента |

Первый экран — 3 параллельных запроса (HTTP/2 их мультиплексирует). Это дешевле
одного монолита, который всё равно пришлось бы дробить пагинацией.

**Лента `upcoming`** объединяет разнородные источники в один отсортированный поток.
Ключ — **дискриминатор `kind`** и общий «конверт», чтобы фронт рендерил единым списком:

```json
{
  "results": [
    {
      "kind": "SUBSCRIPTION_SESSION",
      "date": "2026-06-15", "start_time": "16:00", "end_time": "17:00",
      "student": { "id": 101, "full_name": "Иванов Иван Иванович" },
      "activity_name": "Кружок мышления", "group_name": "Общая",
      "source": { "type": "subscription", "id": 501, "display_id": "#BIG-SUB-777" },
      "is_rescheduled": false
    },
    {
      "kind": "TRIAL",
      "date": "2026-06-17", "start_time": "17:30", "end_time": "18:30",
      "student": { "id": 101, "full_name": "Иванов Иван Иванович" },
      "activity_name": "Английский язык", "group_name": "Начинающие",
      "source": { "type": "trial", "id": 45 }
    },
    {
      "kind": "EVENT",
      "date": "2026-06-20", "start_time": "11:00",
      "title": "Театральные игры",
      "source": { "type": "event", "id": 5 }, "attendees_count": 2
    }
  ],
  "next_cursor": "eyJkYXRlIjoiMjAyNi0wNi0yMCJ9"
}
```

- **Регулярные занятия по абонементу не лежат в БД** — бэкенд генерирует их из слотов
  на горизонт 4 недели (как в Сценарии 2), вплетает пробные/ивенты и сортирует по `date`.
- **Cursor-пагинация, не offset** — лента «живая» (фоновые задачи двигают маски),
  offset поедет; курсор по `(date, id)` стабилен.
- **Без избыточности**: профиль ребёнка в ленте — только `id + full_name`, полная
  карточка живёт в `/me/profile`. Один источник правды на сущность.

### Сценарий 4 — Унифицированная покупка и оплата

**Решение: НЕ полиморфный эндпоинт, а три независимых ресурса.**

```
POST /api/v1/checkout/subscription   (auth required)
POST /api/v1/checkout/trial          (auth required)
POST /api/v1/checkout/event          (анонимно, «быстрая касса»)
```

> Режимы доступа и UX-флоу (2 шага, auth-gate, сохранение intent) — в `auth-flow.md`
> и `checkout-flow.md`. Родитель всегда берётся из токена (для пробного/абонемента) —
> в теле запроса вложенного `parent` нет. Теневая регистрация удалена.

Обоснование против единого `POST /checkout {type, payload}`:

- составы данных **кардинально разные** (абонемент: ребёнок+слоты; пробное: дата+1 слот;
  ивент: число мест) → полиморфный сериализатор выродится в дерево `if type==...`,
  которое невозможно строго типизировать и валидировать;
- **разные политики авторизации** на эндпоинт (401 для абонемента vs аноним для ивента) —
  на полиморфном это рантайм-ветвление вместо декларативного `permission_classes`;
- OpenAPI-схема и кодген клиента для раздельных ресурсов чисты, для полиморфного — «any».

Цена — небольшое дублирование «обёртки заказа», которое выносится в общий сервис
(Шаг 3). Это правильный размен: чистота контракта важнее DRY на уровне URL.

**Request — абонемент** (валидация в `serializers.py`, Шаг 3):

```json
{
  "student_id": 101,
  "new_student": null,
  "plan_id": 3,
  "schedule_ids": [106, 210]
}
```

**Единый Response на все три (`201 Created`)** — здесь унификация уместна, т.к. все
ведут к оплате:

```json
{
  "transaction_id": "a1b2c3d4-e5f6-7890-1234-56789abcdef0",
  "status": "PENDING_PAYMENT",
  "payment_url": "https://yookassa.ru/checkout/payments/...",
  "expires_at": "2026-06-13T16:15:00+07:00"
}
```

Бесплатный ивент (`price: 0`) → `payment_url: null`, `status: CONFIRMED`.

**Архитектурные ответы:**

- **Связь заказа с шлюзом** — на checkout бэкенд: (1) создаёт `transaction(draft)` с
  внутренним UUID, (2) бронирует места (`enrollment.HELD`, `schema-audit §1.1`),
  (3) вызывает `Payment.create` ЮКассы, передавая свой UUID в `metadata` и `idempotence-key`,
  (4) сохраняет `external_payment_id`, переводит `draft → pending`, отдаёт `confirmation_url`.
- **Вебхуки** — см. Шаг 2.2 (отдельный раздел: replay, race conditions, валидация подлинности).

### Сценарий 5 — Журнал посещаемости и списание фишек

Преподаватель открывает журнал на дату+слот и отмечает детей.

```
GET   /api/v1/staff/journal?schedule_id=106&date=2026-06-15
PATCH /api/v1/staff/attendance/{enrollment_id}
```

`GET` отдаёт список записанных детей с текущей отметкой и остатком фишек:

```json
{
  "schedule_id": 106, "date": "2026-06-15",
  "activity_name": "Кружок мышления",
  "students": [
    {
      "enrollment_id": 88, "student": { "id": 101, "full_name": "Иванов Иван Иванович" },
      "status": "NONE",
      "subscription_slot": { "id": 777, "remaining": 3 }
    }
  ]
}
```

**Отметка — `PATCH` (частичное изменение состояния существующей записи), не `POST`:**

```
PATCH /api/v1/staff/attendance/88
Idempotency-Key: 7c9e...        ← обязателен
{ "status": "PRESENT", "date": "2026-06-15" }
```

Ответ `200` возвращает **новый баланс**, чтобы UI не перезапрашивал:

```json
{ "enrollment_id": 88, "status": "PRESENT", "subscription_slot": { "id": 777, "remaining": 2 } }
```

**Архитектурные ответы (транзакционность):**

- Списание/возврат — критическая секция с `SELECT … FOR UPDATE` по строке
  `subscription_slot` (детально — `schema-audit §1.3`). API лишь обёртка над сервисом.
- **`PRESENT` → списание, `ABSENT/NONE` → возврат** выполняются в одной БД-транзакции
  вместе с записью `attendance`; `CHECK(remaining >= 0)` — последний рубеж.
- **Идемпотентность двух уровней**: (1) натуральный ключ `UNIQUE(student, slot, date)`
  не даёт второй фишке списаться при дубль-отметке; (2) `Idempotency-Key` гасит
  сетевой ретрай того же `PATCH` (Шаг 2.1). При конкурентных запросах второй ждёт на
  `FOR UPDATE` и видит уже актуальный баланс.
- `409 SUBSCRIPTION_EXPIRED` — если на момент отметки абонемент просрочен.

## Шаг 2. Безопасность, идемпотентность, вебхуки

### 2.1. Idempotency-Key (checkout + отметки)

Опираемся на IETF draft `Idempotency-Key` и паттерн Stripe.

**Контракт:** клиент шлёт `Idempotency-Key: <uuid4>` на чувствительный `POST/PATCH`.
Логика бэкенда:

1. Завести таблицу `idempotency_record(key PK, request_fingerprint, response_status,
   response_body, created_at)`.
2. На входе — `INSERT key` под уникальным индексом:
   - **новый ключ** → выполняем операцию, сохраняем `status+body` ответа, отдаём клиенту;
   - **ключ существует, тот же `request_fingerprint`** (хэш тела+пути) → отдаём
     **сохранённый ответ** (тот же `status` и `body`), операцию не повторяем;
   - **ключ существует, другой fingerprint** → `409 IDEMPOTENCY_KEY_REUSED` (клиент
     переиспользовал ключ для другого запроса — это его баг).
3. Запись живёт 24 ч, потом чистится (ретраи за пределами суток бессмысленны).

Это превращает «создать платёж» и «списать фишку» в безопасные к повтору операции:
двойной клик / ретрай по таймауту не создадут вторую транзакцию и не спишут вторую фишку.

### 2.2. Вебхуки ЮКассы

```
POST /api/v1/webhooks/yookassa
```

**Контракт валидации подлинности (по рекомендациям ЮКассы):**

1. **HTTPS-only** — эндпоинт принимает только по TLS (ЮКасса по HTTP не шлёт).
2. **IP-allowlist** — пропускать только из диапазонов ЮКассы
   (`185.71.76.0/27`, `185.71.77.0/27`, `77.75.153.0/25`, `77.75.154.128/25`,
   и IPv6 `2a02:5180:...`). Чужой IP → `403`, не раскрывая деталей.
3. **Не доверять телу уведомления.** Тело — лишь сигнал «что-то произошло с платежом X».
   Бэкенд делает **независимый перезапрос** `GET /payments/{id}` к API ЮКассы и берёт
   статус **оттуда**. Это снимает риск подделки тела и устаревших данных.
4. Ответить **`200`** при успешной постановке в обработку — иначе ЮКасса будет слать ретраи.

**Защита от replay / race condition** (см. также `schema-audit §1.1`):

- **Идемпотентность по платежу** — `external_payment_id UNIQUE`; повторный вебхук о том
  же платеже не создаёт второго эффекта.
- **Атомарный переход только из `pending`** —
  `UPDATE transaction SET status='succeeded' WHERE external_payment_id=%s AND status='pending'`;
  0 затронутых строк = уже обработано, идемпотентный no-op.
- **Сетевые задержки / гонка с TTL** — если вебхук `succeeded` пришёл после того, как
  TTL уже перевёл транзакцию в `expired` и снял бронь: сервис реактивирует бронь, а
  если место уже перекуплено — инициирует возврат через API ЮКассы (компенсация).
  Это редкий край, но контракт обязан его описывать.
- **Постановка тяжёлой обработки в очередь** (Taskiq, `public-forms-design §3`) —
  вебхук-вьюха валидирует, ставит задачу и быстро отдаёт `200`; бизнес-логика идёт в воркере.

## Шаг 3. Слои DRF

### 3.1. Сериализаторы

- **Сложные бизнес-правила** — в `validate()` (межполевые) и `validate_<field>()`:
  слот активен; `free > 0`; для пробного — нет существующего пробного по этому кружку
  (`TRIAL_LIMIT_EXCEEDED`); слоты абонемента не конфликтуют по времени между собой;
  `plan.sessions_count` согласован с числом выбранных слотов (4 фишки × слоты).
- **Writable nested «родитель + дети»** (checkout гостя): один сериализатор принимает
  `parent` + `student_id|new_student`. Переопределить `create()`, обернуть в
  `transaction.atomic()`: `Parent.get_or_create(email)`, затем `Student` (новый или
  проверка принадлежности существующего этому родителю — иначе `403 FORBIDDEN_RESOURCE`),
  затем `Subscription/Enrollment/Transaction`. Всё или ничего.
- Чтение и запись — **разные сериализаторы** (`...ReadSerializer` / `...WriteSerializer`):
  на вход узкий валидируемый payload, на выход — денормализованная под рендер структура.

### 3.2. N+1: где болит и что применить (обязательно)

| Эндпоинт | Риск N+1 | Обязательная конструкция ORM |
| -------- | -------- | ---------------------------- |
| `GET /public/schedule` | `activity`, `teacher`, `room` на каждый слот; счётчик мест | `select_related("activity","teacher","room")` + `annotate(taken=Count("enrollments", filter=Q(...)))` |
| `GET /public/activities` | учитель и дни недели по каждому кружку | `select_related("teacher")` + `prefetch_related("slots")` |
| `GET /me/subscriptions` | план + баланс по слотам + данные слотов | `select_related("plan","student")` + `Prefetch("slots", queryset=SubscriptionSlot.objects.select_related("schedule__activity"))` |
| `GET /me/upcoming` | маски на каждую запись | `Prefetch("slot__exceptions", queryset=…filter(date__range=…), to_attr="masks_in_range")` (см. `schema-audit §4`) |
| `GET /staff/journal` | ребёнок + баланс по каждой записи | `select_related("student","subscription_slot")` |

Правило для разработчиков: **FK → `select_related` (JOIN), обратные/M2M →
`prefetch_related` (доп. запрос), агрегаты по связям → `annotate` (а не Python-цикл).**
Любой сериализатор, который в цикле трогает `obj.related.x`, обязан иметь
соответствующий `select_related`/`Prefetch` во вьюсете. Проверять через
`assertNumQueries` в тестах списочных эндпоинтов.

### 3.3. Тонкие вьюсеты
Вьюсет: permissions → разбор/валидация (сериализатор) → вызов сервиса → сериализация
ответа. Деньги, фишки, генерация календаря, процессинг вебхуков — в `services.py`,
не во вьюхах (см. `schema-audit §3.3`).

## Шаг 4. Расширяемость контрактов

Проверка устойчивости к «капризам хозяйки» (волатильность требований из `project-context.md`).

| Будущее изменение | Ломает контракт? | Почему |
| ----------------- | ---------------- | ------ |
| Новые номиналы абонементов (другое число занятий) | **Нет** | Тариф — это строка `subscription_plan`, отдаётся данными, а не зашит в структуру JSON. Добавление номинала = новая запись, контракт неизменен. |
| Групповые скидки | **Минимально** | Цена считается на бэкенде и отдаётся готовой (`total_price`). Скидка — деталь расчёта в `services.py`; можно добавить опциональное поле `discount` в ответ, не ломая существующих клиентов (additive change). |
| Новые типы разовых услуг (SERVICE) | **Нет** | Параметры услуг — в `activity.params JSONB` (`schema-refactoring §A2`). Лента `upcoming` уже полиморфна через `kind` — новый `kind` добавляется аддитивно. |
| Новый способ оплаты | **Нет** | `transaction.metadata JSONB` поглощает специфику провайдера; контракт checkout/вебхука не меняется. |

**Принципы, обеспечивающие это:**
1. **Данные вместо структуры** — варианты (тарифы, типы) приходят строками БД, а не
   формируют разные формы ответа.
2. **Дискриминатор `kind` + общий конверт** в полиморфных лентах — новый вид события
   добавляется без изменения существующих.
3. **Только аддитивные изменения** в мажорной версии: новые **опциональные** поля можно
   добавлять; удаление/переименование/смена типа → `v2`.
4. **JSONB на периферии** (`metadata`, `params`) — поглощает нестабильную форму без
   миграций контракта.
