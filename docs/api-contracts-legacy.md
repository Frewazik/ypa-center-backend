> **Назначение.** Историческая фиксация первых набросков контрактов ЛК и публичной части.
> **Статус.** Архив. Частично устарел: разделы про **авторизацию** и **тела
> checkout-запросов** заменены. Не использовать при реализации без сверки с актуальными.
> **Связанные документы (актуальные).** Авторизация → `auth-flow.md`; UX покупок →
> `checkout-flow.md`; HTTP-контракты → `api-core-contracts.md`. В частности: теневой
> регистрации больше нет, пробное и абонемент — auth-only, родитель из токена, вход по OTP-коду.

# Архитектура авторизации:

> **Раздел отменён.** Описанная ниже схема (теневая регистрация, три режима,
> magic-link) больше не действует. Актуальная модель — в `auth-flow.md`. Текст ниже
> сохранён только как история и не должен использоваться при реализации.

### 1. События (Ивенты) — Режим «Быстрая касса»

- **Требование:** Независимо от авторизации, заполняется с нуля.
- **Как это работает:** Это полностью анонимный флоу. Фронтенд **вообще не отправляет** JWT-токен в заголовках, даже если пользователь залогинен.
- **Бэкенд:** В нашей схеме БД `transaction.parent_id` может быть `null` для ивентов. Мы просто берем имя, телефон, почту, создаем запись `event_registration` и выдаем ссылку на оплату. Никаких привязок к ЛК делать не нужно. Максимально быстрый сбор денег.

### 2. Пробное занятие — Режим «Гибрид» (Теневая регистрация)

Разделяем «быстрые» импульсивные покупки (ивенты) и «осознанные» долгие (абонементы). В e-commerce это называется комбинацией **Guest Checkout** (гостевая покупка) и **Shadow Registration** (теневая регистрация).

- **Требование:** Любой может оформить, но должно падать в профиль. Пользователь регается _в процессе_ покупки.
- **Как это работает:** * **Если пользователь авторизован:** Фронт присылает токен. Бэкенд видит, кто это, и привязывает транзакцию и ребенка к существующему `parent`.

  - **Если это гость:** Фронт присылает только данные формы (без токена). Бэкенд проверяет номер/почту. Если такого нет — **бэкенд сам, в фоне, создает нового `parent`**. После успешной оплаты бэкенд автоматически генерирует Magic Link (ссылку для входа без пароля) и отправляет ее на почту со словами: _"Спасибо за покупку! Ваш личный кабинет создан, вот ссылка для входа"_.

### 3. Абонементы — Режим «Закрытый клуб»

- **Требование:** Только для авторизованных.
- **Как это работает:** Это логика на стороне фронтенда. Когда неавторизованный пользователь нажимает «Собрать абонемент» или «Купить», Кирилл (фронтендер) должен заблокировать форму и показать попап (или перекинуть на страницу): _"Для покупки абонемента войдите в систему"_.
- **Бэкенд:** Эндпоинт покупки абонемента просто вернет `401 Unauthorized`, если туда сунется кто-то без токена.

### 4. Вход в Личный Кабинет (Magic Link)

- **Требование:** Вход по почте.
- **Как это работает:** Пароли не нужны вообще.

  1. Пользователь вводит email.
  2. Бэкенд генерирует одноразовый длинный токен (сохраняет его в таблицу `magic_tokens`) и шлет письмо с кнопкой «Войти в Улицу Радости».
  3. При клике на кнопку пользователь попадает обратно на сайт, фронтенд ловит токен из URL, отправляет на бэкенд, а бэкенд выдает **боевые JWT-токены** (Access/Refresh) для работы с ЛК.

Если разложить логику так, то всё становится очень прозрачным. Бэкенд берет на себя черновую работу (создание юзера при покупке пробного), а фронтенд просто решает, прикреплять заголовок `Authorization` к запросу или нет.
